## 1. 개요

- 담당 — `company-service`, 업체(Company)·상품(Product) 도메인 단독 담당
- 핵심 기술 — Spring Boot, JPA, QueryDSL, PostgreSQL, Redis, Kafka, Spring Cloud(Eureka·Gateway·OpenFeign), Keycloak
- 성격 — 허브·유저·주문 서비스와 동기(Feign)·비동기(이벤트)로 얽히는 MSA 도메인

업체·상품을 관리하고, 주문이 들어올 때 재고를 안전하게 차감하는 서비스를 맡았다. MSA라서 다른 서비스 호출이 실패하거나, 같은 요청이 중복으로 들어오거나, 동시에 재고를 건드리는 상황을 막는 데 집중했다.

---

## 2. 구현 기능

### 업체 (Company)
- 업체 생성 / 조회 / 검색 / 수정 / 삭제 API
- 역할 기반 권한 + 데이터 격리 — MASTER는 전체, HUB_MANAGER는 자기 소속 허브의 업체만, SUPPLIER_MANAGER는 자기 업체만 수정·삭제 가능
- 업체 생성·수정 시 허브 서비스(Feign)로 허브 유효성 검증, 유저 서비스(Feign)로 소유권 확인

### 상품 (Product)
- 상품 생성(허브 활성 검증) / 조회 / 수정 / 삭제 API
- QueryDSL 동적 검색 — 조건(이름·타입 등)에 따라 쿼리를 동적으로 구성
- 재고 확인·차감·복원 내부 API — 주문 서비스와 연동되는 재고 처리 (5번 리팩토링 참고)

## 3. 트러블슈팅 & 기술적 의사결정

각 항목은 `현상 → 원인 분석 → 대안 검토 → 해결 → 결과 → 배운 점` 순서.

### 3-1. 외부 서비스 호출이 DB 트랜잭션을 붙잡는 문제 (Facade 패턴)

- **현상.** 업체 생성·수정 시, 허브 서비스를 호출해 유효성을 검증하는 로직과 DB 저장 로직이 한 서비스 메서드(`@Transactional`)에 섞여 있었다.
- **원인 분석.** 외부 API(Feign) 호출은 네트워크 지연이 생길 수 있는데, 그게 트랜잭션 안에 있으면 DB 커넥션을 잡은 채로 외부 응답을 기다린다. 트래픽이 몰리면 커넥션 풀이 고갈될 위험이 있었다.
- **대안 검토.**
  - ① 그대로 두기 → 커넥션 점유 시간이 길어 위험.
  - ② 트랜잭션 전파 옵션 조정 → 복잡하고 근본 해결이 아님.
  - ③ 검증(외부 호출)과 저장(DB)을 계층으로 분리 → 외부 호출은 트랜잭션 밖에서 먼저, DB 작업만 트랜잭션 안에서. ← 채택
- **해결.** `CompanyFacade`를 두어 권한 검증·범위 검증·허브 검증(Feign)을 트랜잭션 없이 선처리하고, 실제 DB 조작이 필요한 부분만 `CompanyService`(`@Transactional`)로 위임했다.
- **결과.** DB 트랜잭션이 열려 있는 시간이 크게 줄어, 외부 호출 지연이 커넥션 풀에 주는 영향을 없앴다.
- **배운 점.** 트랜잭션은 짧을수록 좋다. 네트워크가 끼는 작업은 트랜잭션 밖으로 빼는 게 커넥션 자원을 지키는 기본이다.

### 3-2. 다른 서비스가 죽으면 내 서비스도 죽는 문제 (Feign Fallback + 데이터 격리)

- **현상.** 업체 검증을 위해 허브·유저 서비스를 Feign으로 호출하는데, 그 서비스가 느리거나 죽으면 내 요청도 그대로 실패·지연됐다.
- **원인 분석.** 동기 호출(Feign)은 상대 서비스 장애가 그대로 전파된다. MSA에서 한 서비스 장애가 연쇄로 번지는 위험이다.
- **대안 검토.** 예외를 그대로 던지기 → 사용자에게 500 노출 / Fallback으로 장애를 격리하고 명확한 에러로 변환 ← 채택
- **해결.** `HubClient`·`UserClient`에 FallbackFactory를 붙여, 호출 실패 시 정해진 대체 응답·에러(`FEIGN_CLIENT_ERROR`)로 변환하도록 했다. 또 인가는 컨트롤러(역할)뿐 아니라 서비스에서 소속 허브·소유 업체 범위까지 검증해, 남의 데이터에 접근하는 것을 막았다.
- **결과.** 의존 서비스 장애가 내 서비스 전체를 무너뜨리지 않고, 실패 지점이 로그에 명확히 남아 원인 파악이 빨라졌다.
- **배운 점.** MSA에서 동기 호출에는 반드시 실패 대비(Fallback)가 필요하다. 그리고 인가는 "역할"만이 아니라 "이 데이터가 이 사람 것인지(범위)"까지 봐야 안전하다.

## 4. 한계와 후속 과제

프로젝트를 마치고 코드리뷰(튜터 피드백)와 회고에서 스스로 짚은 개선 지점이다.

| 한계 | 왜 문제인가 | 후속 과제 |
| --- | --- | --- |
| orderId만으로 멱등성 판단 | 같은 orderId로 다른 내용의 요청이 오면 구분하지 못함 | `orderId + payload 해시` 기반으로 멱등성 키 확장 |
| Redis↔DB 정합성 보정 부재 | Redis 원자 차감 후 DB 반영 사이 장애 시 값이 어긋날 수 있음 (엣지) | 재고 작업 이력 테이블 + 보정 배치로 정합성 재조정 |
| Facade 강결합 | 외부 서비스 동기 호출이 Facade 한 곳에 모여 결합·장애 전파 지점이 됨 | 권한 검증 / 외부 조회 / 유스케이스 조합 책임을 분리 |
| 동기(Feign) 의존 과다 | 서비스 간 동기 호출이 많아 지연·장애에 민감 | 재고/주문/허브 흐름 일부를 Kafka 이벤트 기반으로 재설계 |

## 5. 추가 리팩토링 — 재고 차감을 Redis Lua 원자 처리로 (프로젝트 이후)

> 팀 프로젝트가 끝난 뒤, 코드 리뷰에서 받은 "비관적 락은 대규모에서 병목"이라는 피드백이 계속 남아, 재고 차감 부분을 혼자 다시 손보고 측정까지 해봤다.

- **현상.** 재고 차감을 초기엔 Redis 분산 락 + DB 비관적 락(`FOR UPDATE`) + Redis 완료 키(멱등성)로 구현했다. 코드 리뷰에서 *"대규모인데 재고 차감에 비관적 락을 쓰면 처리량이 병목"*이라는 피드백을 받았다.
- **원인 분석.** `FOR UPDATE`는 한 상품 행에 배타 락을 걸고 트랜잭션 내내 붙잡아, 인기 상품에 요청이 몰리면 직렬화된다. 또 멱등성 완료 키를 커밋 후 별도로 저장해서, 그 사이 Redis 저장이 실패하면 완료 표시가 유실돼 재시도 시 중복 차감 위험이 있었다.
- **대안 검토.** (특강 Redis 2·4장 근거)
  - 낙관적 락(버전) → 경합이 심한 재고엔 재시도가 폭발한다.
  - Redis 원자 명령 / Lua 스크립트 → 락 없이 "확인+차감"을 원자적으로 처리한다. ← 채택 (여러 명령을 묶을 땐 Lua가 정석이고, WATCH보다 경합에 강하다)
- **해결.** 재고 확인·차감·멱등성 기록을 하나의 Lua 스크립트로 원자 처리했다(`ProductStockRedisManager` + `stock-deduct.lua`). Redis가 싱글 스레드라 스크립트 전체가 원자적으로 돌아 락이 필요 없고, 멱등성 기록도 차감과 같은 스크립트 안에서 하므로 "커밋 후 유실" 틈이 사라졌다. Redis만 믿지 않도록 DB를 원천으로 유지해, 성공 시 락 없이 원자 UPDATE(`decreaseStock`)로 반영했다. 비관적 락·분산 락은 제거하고, 취소 시 복원(`restoreStocks`)도 같은 방식으로 멱등 처리했다.
- **결과.** k6로 개선 전/후를 같은 조건(VU 50, 30초, 단일 상품에 동시 차감으로 락 경합 유발)에서 측정했다.

  | 구분 | 처리량 | 평균 | p95 | 성공률 |
  | --- | --- | --- | --- | --- |
  | 비관적 락 (FOR UPDATE) | 165.6 req/s | 299ms | 763ms | 100% |
  | Redis Lua 원자 차감 | 150.7 req/s | 329ms | 592ms | 100% |

  처리량은 개선되지 않았다. 측정 환경이 k6·서비스·PostgreSQL·Redis를 한 PC에서 함께 돌리는 구조라, 처리량이 기계의 CPU 한계(약 165–180 req/s)에 먼저 걸렸다. 즉 이 환경에서는 락이 병목이 아니라 기계가 병목이었고, 락 방식의 차이가 처리량으로 드러나지 않았다. Lua는 요청당 Redis 왕복이 하나 더 붙어 처리량이 오히려 소폭 낮았다. 상위 지연(p90·p95)은 나아졌지만 처리량과 평균은 개선되지 않았다. 다만 단건 검증에서 차감 후 DB와 Redis 재고가 정확히 일치해, 목표였던 원자성·멱등성은 확인했다.
- **배운 점.** 재보기 전엔 락이 병목이라 믿었지만 이 환경에선 아니었다 — 측정 없이 "개선했다"고 말하면 안 된다. 성능을 정량으로 입증하려면 부하 생성기와 서비스를 다른 머신에 분리한 환경이 필요하다. Lua 원자 처리의 확실한 이득은 정합성·멱등성이었고, 처리량 이점은 부하가 분산된 실제 인프라의 몫이라 이번엔 정량 입증까지는 못 했다. 그래서 이 작업의 결론은 "몇 배 빨라졌다"가 아니라 "구조를 바꾸고, 재보고, 환경의 한계를 이해했다"이다.

> 남은 과제 — 부하 생성기와 서비스를 다른 머신에 분리하고 DB 리소스를 확보한 환경에서 재측정하면, 락 제거의 처리량 이점을 정량으로 확인할 수 있을 것이다.
